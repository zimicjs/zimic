# Getting started

This guide will help you get started with `@zimic/interceptor`. The quick start uses HTTP interceptors first. For
WebSocket mocks, see the [WebSocket interceptor guide](/docs/interceptor/guides/ws/local-interceptors) and
[API reference](/docs/interceptor/api).

## Requirements

HTTP interception requires the [Fetch API](https://developer.mozilla.org/docs/Web/API/Fetch_API). Browser clients must
provide Fetch natively or through a polyfill. Server-side HTTP interception requires Node.js 22 or later.

WebSocket interception requires a native or polyfilled
[WebSocket API](https://developer.mozilla.org/docs/Web/API/WebSocket) in every client runtime. Browsers normally provide
it natively; Node.js clients must expose the native `WebSocket` or a compatible polyfill. Node.js projects consuming the
WebSocket interceptor packages, including interceptor servers, require Node.js 22.4 or later.

### Supported environments

<Tabs groupId="runtime-environments">
  <TabItem value="client" label="Client side">

`@zimic/interceptor` supports HTTP clients with Fetch and WebSocket clients with a native or polyfilled `WebSocket`.
This includes relatively modern browsers. Local browser interceptors also require an initialized mock service worker;
the protocol guides describe that setup. [Can I Use](https://caniuse.com/?search=fetch) is a great resource to check the
compatibility of specific features with different browsers.

:::info INFO: <span>Optional protocol peers</span>

Note that [`@zimic/http`](/docs/http) and [`@zimic/ws`](/docs/ws) are peer dependencies of `@zimic/interceptor`. Install
each or both depending on your needs. HTTP examples use `@zimic/http` schemas, while
[WebSocket examples](/docs/interceptor/guides/ws/local-interceptors) use [`@zimic/ws` schemas](/docs/ws/guides/schemas).
When upgrading `@zimic/interceptor` to a new version, consider upgrading both schema packages as well to ensure that the
versions are compatible.

The protocol peers are optional, so single-protocol projects can install only the schema package they use: `@zimic/http`
for HTTP interceptors, `@zimic/ws` for WebSocket interceptors, or both when a project uses both protocols.

:::

## Your first HTTP interceptor

### Declaring an HTTP schema

To start using `@zimic/interceptor`, declare an HTTP schema using [`@zimic/http`](/docs/zimic-http/guides/1-schemas.md).
The schema represents the structure of your API, including the paths, methods, request and response types.

:::tip TIP: <span>OpenAPI Typegen</span>

For APIs with an [OpenAPI documentation](https://www.openapis.org) (e.g. [Swagger](https://swagger.io)), the
[`zimic-http typegen` CLI](/docs/zimic-http/guides/2-typegen.mdx) can automatically infer the types and generate the
schema for you. This is a great way to keep your schema up to date and save time on manual type definitions.

:::

```ts title='schema.ts'
import { HttpSchema } from '@zimic/http';

interface User {
  username: string;
}

interface RequestError {
  code: string;
  message: string;
}

type Schema = HttpSchema<{
  '/users': {
    POST: {
      request: { body: User };
      response: {
        201: { body: User };
        400: { body: RequestError };
        409: { body: RequestError };
      };
    };

    GET: {
      request: {
        headers: { authorization: string };
        searchParams: {
          query?: string;
          limit?: number;
        };
      };
      response: {
        200: { body: User[] };
        400: { body: RequestError };
        401: { body: RequestError };
      };
    };
  };

  '/users/:userId': {
    PATCH: {
      request: {
        headers: { authorization: string };
        body: Partial<User>;
      };
      response: {
        204: {};
        400: { body: RequestError };
      };
    };
  };
}>;
```

### Creating an HTTP interceptor

With the schema defined, you can now create an [HTTP interceptor](/docs/interceptor/api/http-interceptor).

`@zimic/interceptor` provides a [`createHttpInterceptor`](/docs/interceptor/api/create-http-interceptor) function that
takes the schema as a type parameter and returns an interceptor instance. It allows you to intercept HTTP requests,
validate their contents, and return mock responses, all automatically typed based on the schema. The `baseURL` option
represents the scope of the interceptor and points to the URL that your application will use to make requests.

```ts
import { createHttpInterceptor } from '@zimic/interceptor/http';

const interceptor = createHttpInterceptor<Schema>({
  baseURL: 'http://localhost:3000',
});
```

You can also set other options, such as the interceptor type and how unhandled requests should be treated. Refer to the
[`createHttpInterceptor` API reference](/docs/interceptor/api/create-http-interceptor) for more details.

:::info INFO: <span>Choosing your interceptor type</span>

HTTP interceptors are available in two types: `local` (default) and `remote`.

<Tabs groupId="zimic-http-interceptor-type">
  <TabItem value="local" label="Local interceptor">

```ts
import { createHttpInterceptor } from '@zimic/interceptor/http';

const interceptor = createHttpInterceptor<Schema>({
  // highlight-next-line
  type: 'local',
  baseURL: 'http://localhost:3000',
});
```

  </TabItem>

  <TabItem value="remote" label="Remote interceptor">

```ts
import { createHttpInterceptor } from '@zimic/interceptor/http';

const interceptor = createHttpInterceptor<Schema>({
  // highlight-next-line
  type: 'remote',
  baseURL: 'http://localhost:3000',
});
```

  </TabItem>
</Tabs>

When an interceptor is `local`, Zimic uses [MSW](https://github.com/mswjs/msw) to intercept requests _in the same
process_ as your application. This is the simplest way to start mocking requests and does not require any server setup.

Interceptors with type `remote` use a dedicated [interceptor server](/docs/interceptor/cli/server) to handle requests.
This opens up more possibilities for mocking, such as handling requests from multiple applications. It is also more
robust because it uses a regular HTTP server and does not depend on local interception algorithms.

Learn more about [local interceptors](/docs/interceptor/guides/http/local-interceptors) and
[remote interceptors](/docs/interceptor/guides/http/remote-interceptors) to see which one fits your needs. We recommend
starting with local interceptors, as they are easier to get started with, and moving to remote interceptors if your use
case is best served by them.

:::

## HTTP interceptor lifecycle

### Starting an interceptor

To intercept requests, an interceptor must be started with
[`interceptor.start()`](/docs/interceptor/api/http-interceptor#interceptorstart). This is usually done in a `beforeAll`
hook in your test suite.

```ts
beforeAll(async () => {
  // highlight-next-line
  await interceptor.start();
});
```

:::info INFO: <span>Local interceptors in browsers</span>

If you are using a [local interceptor](/docs/interceptor/guides/http/local-interceptors) in a **browser** environment,
you must first [initialize a mock service worker](/docs/interceptor/cli/browser#zimic-interceptor-browser-init) in your
public directory before starting the interceptor.

:::

:::info INFO: <span>Remote interceptors and interceptor servers</span>

If you are using a [remote interceptor](/docs/interceptor/guides/http/remote-interceptors), the `baseURL` should point
to a running [interceptor server](/docs/interceptor/cli/server), which is configured by the interceptor to handle
requests. Learn more about
[starting remote HTTP interceptors](/docs/interceptor/guides/http/remote-interceptors#starting-an-interceptor).

:::

### Clearing an interceptor

When using an interceptor in tests, it's important to clear it between tests to avoid that one test affects another.
This is performed with [`interceptor.clear()`](/docs/interceptor/api/http-interceptor#interceptorclear), which resets
the interceptor and handlers to their initial states.

<Tabs groupId="zimic-http-interceptor-type">
  <TabItem value="local" label="Local interceptor">

```ts
beforeEach(() => {
  // highlight-next-line
  interceptor.clear();
});
```

  </TabItem>

  <TabItem value="remote" label="Remote interceptor">

```ts
beforeEach(async () => {
  // highlight-next-line
  await interceptor.clear();
});
```

  </TabItem>
</Tabs>

### Checking expectations

After each test, you can check if your application has made all of the expected requests with
[`interceptor.checkTimes()`](/docs/interceptor/api/http-interceptor#interceptorchecktimes). Learn more about how
interceptors support [declarative assertions](/docs/interceptor/guides/http/declarative-assertions) to keep your tests
clean and readable.

<Tabs groupId="zimic-http-interceptor-type">
  <TabItem value="local" label="Local interceptor">

```ts
afterEach(() => {
  // highlight-next-line
  interceptor.checkTimes();
});
```

  </TabItem>

  <TabItem value="remote" label="Remote interceptor">

```ts
afterEach(async () => {
  // highlight-next-line
  await interceptor.checkTimes();
});
```

  </TabItem>
</Tabs>

### Stopping an interceptor

After the interceptor is no longer needed, such as at the end of your test suite, you can stop it with
[`interceptor.stop()`](/docs/interceptor/api/http-interceptor#interceptorstop).

```ts
afterAll(async () => {
  await interceptor.stop();
});
```

## Mocking requests

You can now use the interceptor to handle requests and return mock responses. All paths, methods, parameters, requests,
and responses are typed by default based on the schema.

<Tabs groupId="zimic-http-interceptor-type">
  <TabItem value="local" label="Local interceptor">

```ts
test('example', () => {
  const users: User[] = [{ username: 'me' }];

  //  highlight-start
  interceptor
    .get('/users')
    .with({
      headers: { authorization: 'Bearer my-token' },
      searchParams: { query: 'u' },
    })
    .respond({
      status: 200,
      body: users,
    })
    .times(1);
  //  highlight-end

  // Run the application and make requests...
});
```

  </TabItem>

  <TabItem value="remote" label="Remote interceptor">

```ts
test('example', () => {
  const users: User[] = [{ username: 'me' }];

  //  highlight-start
  await interceptor
    .get('/users')
    .with({
      headers: { authorization: 'Bearer my-token' },
      searchParams: { query: 'u' },
    })
    .respond({
      status: 200,
      body: users,
    })
    .times(1);
  //  highlight-end

  // Run the application and make requests...
});
```

  </TabItem>
</Tabs>

:::info INFO: <span>Remote interceptors are asynchronous</span>

Many operations in [remote interceptors](/docs/interceptor/guides/http/remote-interceptors) are **asynchronous** because
they may involve communication with an [interceptor server](/docs/interceptor/cli/server). This is different from
[local interceptors](/docs/interceptor/guides/http/local-interceptors), which have mostly **synchronous** operations.

:::

If you need to access the requests processed by the interceptor, use
[`handler.requests`](/docs/interceptor/api/http-request-handler#handlerrequests).

<Tabs groupId="zimic-http-interceptor-type">
  <TabItem value="local" label="Local interceptor">

```ts
// highlight-next-line
const handler = interceptor
  .get('/users')
  .with({
    headers: { authorization: 'Bearer my-token' },
    searchParams: { query: 'u' },
  })
  .respond({
    status: 200,
    body: users,
  })
  .times(1);

// Run the application and make requests...

//  highlight-start
console.log(handler.requests.length); // 1

console.log(handler.requests[0].headers.get('authorization')); // 'Bearer my-token'

console.log(handler.requests[0].searchParams.size); // 1
console.log(handler.requests[0].searchParams.get('username')); // 'my'

console.log(handler.requests[0].body); // null
//  highlight-end
```

  </TabItem>

  <TabItem value="remote" label="Remote interceptor">

```ts
// highlight-next-line
const handler = await interceptor
  .get('/users')
  .with({
    headers: { authorization: 'Bearer my-token' },
    searchParams: { query: 'u' },
  })
  .respond({
    status: 200,
    body: users,
  })
  .times(1);

// Run the application and make requests...

//  highlight-start
console.log(handler.requests.length); // 1

console.log(handler.requests[0].headers.get('authorization')); // 'Bearer my-token'

console.log(handler.requests[0].searchParams.size); // 1
console.log(handler.requests[0].searchParams.get('username')); // 'my'

console.log(handler.requests[0].body); // null
//  highlight-end
```

  </TabItem>
