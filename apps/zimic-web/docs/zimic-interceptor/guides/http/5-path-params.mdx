# Using path params

Path parameters are a way to include dynamic values in the URL path of a request. They are typically used to identify a
resource being accessed or modified. For example, in the URL `/users/:userId`, `:userId` represents a path parameter
that can be replaced with a specific user identifier when making a request.

## Using request path params

Path params are automatically inferred from the paths in your [schema](/docs/zimic-http/guides/1-schemas.md). To define
a path param, prefix it with a colon (`:`) in the path string. See
[Declaring paths](/docs/zimic-http/guides/1-schemas.md#declaring-paths) for more details on how to declare required,
optional, and repeating path params.

```ts title='schema.ts'
import { HttpSchema } from '@zimic/http';

interface User {
  id: string;
  username: string;
}

type Schema = HttpSchema<{
  // highlight-next-line
  '/users/:userId': {
    PATCH: {
      request: {
        body: Partial<User>;
      };
      response: {
        204: {};
      };
    };
  };
}>;
```

Then, the path params of the requests will be available in the `pathParams` property, such as in
[`handler.with()`](/docs/zimic-interceptor/api/3-http-request-handler.mdx#handlerwith),
[computed responses](/docs/zimic-interceptor/api/3-http-request-handler.mdx#handlerrespond), and
[`handler.requests`](/docs/zimic-interceptor/api/3-http-request-handler.mdx#handlerrequests).

You can use path params directly in the path string using template literals. `@zimic/interceptor` automatically infers
the endpoint and types all resources as usual.

### Matching any value

Interceptors accept path params in the path string, declared with a colon (`:`) as in the schema. This allows you to
match any value for that parameter. If you need to escape a colon in the path, use two backslashes before the `:`
(`\\:`).

By default, path params match a single segment, not including slashes (`/`). See
[Declaring paths](/docs/zimic-http/guides/1-schemas.md#declaring-paths) for more details on how to declare repeating
path params, which accept multiple segments.

<Tabs groupId="zimic-http-interceptor-type">
  <TabItem value="local" label="Local interceptor">

```ts
const handler = interceptor
  // Match any user id
  // highlight-next-line
  .patch('/users/:userId')
  .respond((request) => {
    // highlight-next-line
    console.log(request.pathParams.userId);

    return { status: 204 };
  })
  .times(1);

// Run the application and make requests...

console.log(handler.requests); // 1
// highlight-next-line
console.log(handler.requests[0].pathParams.userId);
```

  </TabItem>

  <TabItem value="remote" label="Remote interceptor">

```ts
const handler = await interceptor
  // Match any user id
  // highlight-next-line
  .patch('/users/:userId')
  .respond((request) => {
    // highlight-next-line
    console.log(request.pathParams.userId);

    return { status: 204 };
  })
  .times(1);

// Run the application and make requests...

console.log(handler.requests); // 1
// highlight-next-line
console.log(handler.requests[0].pathParams.userId);
```

  </TabItem>
</Tabs>

Use [`interceptor.with()`](/docs/zimic-interceptor/api/3-http-request-handler.mdx#handlerwith) in case you need
validation or to limit a parameter to specific values.

<Tabs groupId="zimic-http-interceptor-type">

```ts
interceptor
  // Match any numeric user id
  // highlight-start
  .patch('/users/:userId')
  .with((request) => /^\d+$/.test(request.pathParams.userId))
  // highlight-end
  .respond({ status: 204 })
  .times(1);
```

  </TabItem>
